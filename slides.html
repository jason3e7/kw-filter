真實情境：工程師請 AI 審查程式碼

❌ 直接貼上
# config.py
DB_HOST     = "prod-db.acme-corp.com"

✅ 用 kw-filter 處理後
# config.py
DB_HOST     = "prod-db.[[KW_A1B2C3D4]].com"
DB_USER     = "admin"
DB_PASSWORD = "[[KW_E5F6A7B8]]"
API_KEY     = "[[KW_C9D0E1F2]]"

OWNER_EMAIL = "[[KW_33445566]]"
PROJECT     = "[[KW_77889900]]"

AI 仍然能理解程式碼結構、提供重構建議——但不會看到任何真實的機密值。

運作原理：5 個獨立指令

search  先偵查

掃描目錄，列出關鍵字出現的檔案、行號、欄位。支援遞迴與 JSON 輸出。

clear  直接清空

將關鍵字替換為空字串或自訂文字。適合不需要還原的場景。

replace  換成 token

每個關鍵字換成 [[KW_XXXX]]，同一關鍵字跨檔案共用同一 token，並輸出 mapping.json。

restore  還原

讀取 mapping.json，將 token 換回原始值。不需要原始關鍵字清單。

cleanlog  清理日誌

整行刪除含關鍵字的行。支援 --dry-run 預覽與 --stats 統計比例。

案例一：法律文件 AI 摘要

情境

律師事務所每週需要請 AI 摘要 30–50 份合約，合約內含客戶姓名、統編、金額、條款細節。

問題

合約是客戶機密，直接貼上 ChatGPT 違反保密義務，且違反部分國家的資料保護法規（如 GDPR、個資法）。

解法

維護 keywords.txt，執行 replace 再送 AI；拿回摘要後執行 restore 還原。

送出前（原始）
甲方：台灣科技股份有限公司
代表人：陳建宏
統一編號：12345678
合約金額：新台幣 500 萬元
生效日期：2024-03-01
送出後（token 化）
甲方：[[KW_A1B2C3D4]]
代表人：[[KW_E5F6A7B8]]
統一編號：[[KW_C9D0E1F2]]
合約金額：新台幣 [[KW_33445566]] 元
生效日期：2024-03-01

AI 仍能摘要條款結構、義務事項、違約責任——機密數值完全遮蔽。

案例二：程式碼庫 AI 重構

情境

後端團隊想用 AI 重構一個有 200+ 個 Python 檔的專案，但 repo 中散落著：

  • 硬編碼的 API Key
  • 內部服務 domain（api.internal.acme.com）
  • 資料庫連線字串
  • 員工姓名寫在 TODO 裡

執行方式

python3 kw_tools.py search \
  -k secrets.txt -t ./src -r

python3 kw_tools.py replace \
  -k secrets.txt -t ./src -r -m mapping.json

python3 kw_tools.py restore \
  -m mapping.json -t ./refactored -r

效果

✅ AI 看到完整的邏輯結構、函式命名、模組關係

✅ 所有機密字串以 token 形式保留，不影響 AI 理解程式流程

✅ 還原後，重構結果直接可用，不需要手動把 API Key 貼回去

📊 實測：100 個檔案 / 10 萬行 ⟶ 處理時間 0.34 秒

案例三：客服日誌 AI 分析

情境

電商公司每月有 5 萬筆客服對話紀錄，想用 AI 分析常見問題類型與情緒趨勢，但日誌中含有：

  • 用戶姓名、Email
  • 訂單編號（可反查個資）
  • 客服人員姓名
  • 地址片段

workflow

① search 確認覆蓋率（有沒有漏掉的關鍵字）
② replace 批次處理 5 萬筆日誌
③ 送 AI 做分類、情緒分析、趨勢報告
④ restore 報告中如有引用個案，還原姓名

關鍵字清單範例

# customer_kw.txt
# 常見姓名
王小明 / 陳美麗 / 李大偉
# 客服人員
服務專員 Andy
服務專員 Sarah
# 訂單前綴
ORD-2024-

符合個人資料保護法第 6 條「去識別化」處理要求，可作為合規佐證。

案例四：系統日誌清理 （新功能）

情境

DevOps 工程師需要把 production log 送給 AI 找出異常模式，但日誌中夾雜：

  • 用戶 Email / 帳號 ID
  • IP 位址（內部拓樸資訊）
  • 含機密值的 debug 行（token=xxx）
  • 認證錯誤行（含嘗試的密碼明文）

這類日誌「部分遮蔽」不夠——只要含有機密，整行都需要移除。

cleanlog 與 clear 的差異

clear：token=secret → token=（殘留結構）
cleanlog：整行刪除，不留任何痕跡

原始 log（含機密）
10:00:01 INFO  Server started
10:00:05 INFO  Login: alice@corp.com
10:01:22 ERROR Auth fail: secret_token invalid
10:01:23 INFO  Retry connection...
10:02:00 ERROR Timeout on 192.168.1.100
10:02:01 INFO  Connection restored
10:03:10 DEBUG token=sk-live-abc123 cached
cleanlog 處理後
10:00:01 INFO  Server started
10:01:23 INFO  Retry connection...
10:02:01 INFO  Connection restored
# --dry-run 先預覽，再執行
python3 kw_tools.py cleanlog \
  -k keywords.txt -t ./logs -r --dry-run
# → 4 line(s) would be removed

python3 kw_tools.py cleanlog \
  -k keywords.txt -t ./logs -r --stats
# → 4 removed / 3 kept (57.1%)

驗證二：效能實測

Regex 編譯次數

為什麼快？

  • 所有關鍵字合併成 單一 Regex，只掃描一次
  • 關鍵字依長度排序，最長優先，避免重複回溯
  • Binary search（bisect）做 O(log n) 關鍵字存在確認
  • 逐行 iterator，不一次載入整個大檔

測試環境

  • Python 3.10 · Ubuntu 20.04
  • 4 個關鍵字，每行平均含 2 個命中
  • 100 個檔案 × 1,000 行 = 100,000 行
  • 無 SSD 加速（標準 VM 磁碟）

實際企業文件通常每行命中率更低，速度會更快。

驗證三：正確性保證

最長優先

關鍵字 John 和 John Doe 同時存在時，John Doe 優先被匹配。

Input:  John Doe was here
Output: [[KW_AAAA]] was here
✓ 只有 1 個 token（非 2 個）

穩定 Token

同一關鍵字在所有檔案中對應同一個 token，還原後保持語意一致性。

file_a.txt: [[KW_AAAA]]
file_b.txt: [[KW_AAAA]]
→ restore → 兩者都還原為 "John Doe"

Unicode 完整支援

繁體中文、日文、特殊符號均正確處理，不會造成字元截斷或亂碼。

Input:  作者：張三
Output: 作者：[[KW_BBBB]]
restore → 作者：張三 ✓

🔁 Roundtrip 保證：replace → restore 後，所有測試案例均通過逐字元比對（含多檔案、Unicode、多次出現同一關鍵字）。
